# 电商售后材料管理工具 — 设计文档

**日期**:2026-06-15
**状态**:已确认,待编写实现计划

---

## 1. 概述

一个**桌面端**电商售后材料管理工具,用于整理、归档、检索售后场景的视频与图片素材,并能快速打包发给平台客服。

- **定位**:轻量级的本地素材整理/取证工具。**不需要**防篡改、哈希校验、可信时间戳等司法级取证能力。
- **平台**:Windows + macOS。
- **数据**:全部存本地硬盘,**数据根目录用户可更改/迁移**。
- **规模**:百级售后单、累计几十 GB、**单机单用户**,无性能瓶颈压力。

---

## 2. 技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 外壳框架 | **Electron** | 生态成熟;内置 Chromium 保证视频播放在 Win/Mac **表现一致、格式兜底最好**(售后视频格式杂);文件/zip/缩略图工具链现成 |
| 前端 | React + TypeScript + Tailwind CSS | UI 开发效率高 |
| 元数据库 | SQLite(`better-sqlite3`) | 嵌入式、单文件、可靠;用 **FTS5** 做多单号全文检索 |
| 图片缩略图 | `sharp` | 成熟高效 |
| 视频抽帧 | `ffmpeg`(打包静态二进制) | 生成视频缩略图 / 截帧 |
| 打包 | `archiver`(zip) | 导出整单或勾选材料 |
| 构建分发 | `electron-builder` | 出 Win/Mac 安装包 |

> 备选方案 Tauri 因 macOS 系统 WebView(WKWebView)对部分视频格式支持不足而放弃;Flutter Desktop 因桌面媒体/文件生态不成熟放弃。

---

## 3. 架构

Electron 标准三层进程模型:

- **主进程(Main)**:所有"碰硬盘"的操作——文件系统读写、SQLite、ffmpeg/sharp 缩略图、zip 打包、设置(数据根目录)管理。
- **预加载(Preload)**:用 `contextBridge` 暴露一组受控 API(如 `importFiles`、`searchTickets`、`exportZip`)给渲染层。**不开启 `nodeIntegration`**,保证安全。
- **渲染进程(Renderer)**:React + TS + Tailwind,纯 UI,经 IPC 调用主进程。

### 设计原则
- **关注点隔离**:UI 层不直接接触文件系统与数据库;所有副作用集中在主进程,便于独立测试。
- **人类可读优先**:磁盘上的文件夹结构本身就能让人不开应用也能找到材料。
- **可重建**:DB 仅作索引/缓存,丢失后可由文件夹结构重新扫描重建。

## 4. 数据模型与存储布局

### 4.1 磁盘布局

```
<数据根目录>/
  2024-0001/              # 以售后单号命名的文件夹
    images/               # 原始图片(保留原文件名,冲突时加序号后缀)
    videos/               # 原始视频
  2024-0002/
    ...
  .thumbnails/            # 缩略图缓存(可重建,不属于原始数据)
  vhelper.db              # SQLite 元数据库
```

- **组织单位**:售后单号(成案单位)。
- 原始文件保留原文件名;同名冲突时追加序号。

### 4.2 SQLite 表

**`tickets`** — 售后单
- `aftersale_no`(主键,售后单号)
- `order_no`(订单号)
- `shipping_no`(发货单号)
- `return_no`(退货单号)
- `status`(状态:待处理/处理中/已解决 等)
- `note`(备注)
- `created_at` / `updated_at`

**`materials`** — 材料文件
- `id`(主键)
- `aftersale_no`(外键 → tickets)
- `rel_path`(相对数据根目录的路径)
- `kind`(image / video)
- `captured_at`(原始拍摄时间,来自 EXIF,可空)
- `imported_at`(导入时间)
- `size_bytes`
- `thumb_path`(缩略图相对路径,可空)

**`tickets_fts`** — FTS5 全文索引
- 覆盖 `aftersale_no` / `order_no` / `shipping_no` / `return_no` / `note`,支持任意单号片段即时检索。

### 4.3 数据根目录迁移
- 设置中更改根目录:引导将现有文件 + DB 整体搬到新位置,或指向一个已存在的库。
- DB 内只存**相对路径**,因此整个数据文件夹可随意拷贝/换盘而不失效。

## 5. 功能模块

1. **售后单管理**:新建/编辑售后单(售后单号、关联订单号/发货单号/退货单号、状态、备注);列表展示。
2. **材料导入**:拖拽文件/文件夹到某售后单,或"导入到新建售后单";自动按图片/视频分类并生成缩略图。
3. **检索**:顶部搜索框,输入任意单号片段(售后/订单/发货/退货)即时定位;支持按状态、时间筛选。
4. **预览**:网格缩略图 → 点开大图轮播 / 视频播放器;视频支持**截帧另存**。
5. **输出**:
   - 单文件"复制到剪贴板"(图片可直接粘贴给客服聊天窗口)
   - "导出到文件夹"(按单号/时间重命名)
   - "打包 zip"(整单或勾选的材料)

### 典型流程
新客诉 → 新建售后单填单号 → 拖入聊天截图/质检视频 → 标状态"处理中" + 备注 → 需举证时搜单号 → 勾材料 → 打包 zip 发客服 → 标状态"已解决"。

---

## 6. 错误处理

- **导入异常**(文件被占用/损坏/超大):跳过该文件并提示,不中断整批导入。
- **缩略图生成失败**(如 ffmpeg 解不了的格式):回退占位图,不影响原文件保存与播放。
- **根目录不可写 / 磁盘满**:导入前预检,明确报错。
- **DB 与文件不一致**(用户在应用外手动删改文件):提供"重新扫描 / 校准索引"功能,以文件夹为准重建索引。

---

## 7. 测试策略

- **主进程核心逻辑**(导入、检索、zip 打包、根目录迁移):优先 **TDD**,单元测试覆盖。
- **数据库层**:临时库测试增删查与 FTS 检索。
- **渲染层**:关键交互做轻量组件测试,不追求覆盖率。

---

## 8. 明确不做(YAGNI)

- 不做哈希校验 / 可信时间戳 / 防篡改链(用户明确不需要)。
- 不做 PDF 取证报告(输出只需复制/导出文件夹/zip)。
- 不做多用户、云同步、服务端(单机本地)。
- 不为大规模(万级以上)做特殊性能架构(当前百级即可,结构上不阻碍未来扩展)。
